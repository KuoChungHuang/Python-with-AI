## 1. 專案目標與三階段範圍

| 階段 | 目標 | 核心產出 |

| 第一階段 | 建立可作答的練習測驗網站，功能對齊兩個參考網站 | 分類練習、隨機測驗、答對/答錯判定 |

## 4. 系統架構總覽

### 4.1 技術路線建議

使用者尚未指定技術路線，且第三階段需要呼叫 Claude API（金鑰不可放在純前端），因此建議採用「分層漸進」架構，避免第一階段選錯路線導致第三階段重寫：

1. **第一、二階段**：純前端靜態網站（建議 Vite + React 或 Vue，任一皆可，視使用者熟悉度）＋ 本機 JSON 題庫檔案。不需資料庫、不需伺服器，可在本機直接開發測試，之後如需公開再選擇 GitHub Pages / Vercel 等靜態託管。
2. **第三階段**：新增一個輕量後端（建議 Vercel/Cloudflare 的 Serverless Function，或簡易 Node.js API），專責保管 Claude API 金鑰、執行檢索（RAG）與轉發問答請求。前端其餘部分不需改動。

此路線的關鍵原則：**前端不要直接呼叫任何需要金鑰的外部 API**，所有「問答」邏輯集中在後端服務層，第一、二階段完全不受影響。

### 4.2 分層架構

```
UI 層（畫面／互動）
   │
資料存取層（Repository：目前讀本機 JSON，未來可換成 API 呼叫，UI 層不需感知差異）
   │
資料層
 ├─ Phase 1：questions.json（純題目與答案）
 ├─ Phase 2：questions.json 擴充 explanation／legal_refs 欄位
 └─ Phase 3：法規全文語料庫 ＋ 後端問答服務
```

「資料存取層」是三階段銜接的關鍵：第一階段就用固定介面（例如 `getQuestionsByCategory()`、`getRandomQuestions(n)`）包裝資料讀取，即使未來資料來源從 JSON 換成 API，UI 元件呼叫方式不變。

## 5. 資料模型設計

### 5.1 第一階段題目結構

```json
{
  "id": "eng-001",
  "category": "工程及技術服務採購作業",
  "type": "single",        // single=四選一, tf=是非題
  "stem": "下列何項費用，不得納入建造費用計算技術服務費用？",
  "options": {
    "A": "承包商的利潤",
    "B": "承包商之棄土費",
    "C": "物價指數調整工程款",
    "D": "承包商於工地開挖出有價料之減項金額"
  },
  "answer": "C"
}
```

`options` 改用 A/B/C/D 字串鍵值取代陣列索引，避免前後端對接或撰寫 AI 提示詞時因 0-based 索引與題目慣用的 1-4／A-D 標示不一致而產生誤判。

是非題（`type: "tf"`）不需要 `options`，`answer` 改為 `true`/`false`。

### 5.2 第二階段擴充欄位

```json
{
  "...": "沿用第一階段欄位",
  "explanation": "說明為何該選項正確、其他選項錯在哪裡",
  "legal_refs": [
    { "law": "政府採購法", "article": "第22條第1項第9款", "url": "https://law.moj.gov.tw/..." },
    { "law": "政府採購法施行細則", "article": "第22條第4項" }
  ],
  "source_note": "參考：政府採購稽核發現缺失實例彙編 貳、四、11"
}
```

### 5.3 第三階段補充結構

- 法規語料庫（供檢索用）：切塊粒度細化至「項／款」，而非整條法條。政府採購法多有長條文、多款項（如第22條第1項即有16款），若以整條為最小檢索單位，容易讓 RAG 回覆失焦、引用不精確；細化至項/款可提升檢索精確度。每塊記錄法規名稱、條號、項次、款次、條文全文、（若可取得）相關函釋摘要，並預留 `embedding` 向量欄位供未來導入向量檢索使用。此為 Phase 3 技術選型確認後的設計備忘，現階段不需實作。
- 對話紀錄結構：問題、Claude 回覆、引用來源清單、時間戳記（僅本機保存，不涉及使用者帳號）。

## 6. 第一階段規格：練習測驗網站

### 6.1 功能清單

1. 首頁：14 大類主題選單＋「隨機測驗」「模擬考」入口。
2. 分類練習：選定一類後，逐題作答，答題後立即顯示對錯（本階段尚不顯示解析）。
3. 隨機測驗：從全題庫或指定類別中隨機抽題，數量可選（如 10 / 20 / 50 題）。
4. 模擬考模式：固定題數＋倒數計時，作答完畢顯示成績與逐題對錯。
5. 錯題本：作答錄下的錯題自動加入清單，可針對錯題本重新練習。
6. 上一題／下一題導覽，並顯示目前進度（第 x／y 題）。
7. 字體大小設定（基本無障礙需求）。
8. 作答狀態（目前進度、錯題本）以瀏覽器本機儲存（localStorage）保存，不需登入。

### 6.2 頁面規劃

- `/`：首頁（分類選單、模式入口）
- `/quiz/:category`：分類練習頁
- `/quiz/random`：隨機測驗頁
- `/exam`：模擬考頁（計時）
- `/review`：錯題本頁
- `/result`：測驗結果頁（本次測驗結束後顯示）

### 6.3 資料處理流程：PDF 轉結構化題庫

1. 以 `pdftotext -layout` 將 `20260625全部題庫.pdf` 轉為文字。
2. 撰寫解析腳本（Python），依「行首縮排 0–3 字元＋題號＋答案代號」的固定格式，切出每一題的題號、答案、題幹與選項。
3. 依「選擇題」／「是非題」子標題切分題型；依 14 個主題標題切分類別。
4. 解析結果輸出為 `questions.json`，並人工抽樣核對至少每類別 5 題，確認選項與答案對應正確。
5. 將 `questions.json` 放入前端專案的資料目錄，由資料存取層讀取。

### 6.4 開發任務清單

1. 建立前端專案骨架（框架、路由、狀態管理）。
2. 撰寫並執行 PDF 解析腳本，產出 `questions.json`。
3. 實作資料存取層（`getQuestionsByCategory`、`getRandomQuestions`、`getQuestionById` 等）。
4. 實作分類練習頁與單題作答元件。
5. 實作隨機測驗與模擬考模式（含計時器）。
6. 實作錯題本（localStorage 讀寫）。
7. 實作測驗結果頁（成績、逐題對錯清單）。
8. 版面與字體設定、基本行動裝置版面調整。
9. 以實際題庫跑過全部 14 類別，確認題目顯示與判分正確。

### 6.5 驗收標準

- 每一類別至少可完整跑完一輪練習，題目與選項與 PDF 原文一致。
- 隨機測驗、模擬考計時、錯題本三項功能皆可正常運作。
- 重新整理頁面後，錯題本與進度不因此消失（本機儲存生效）。

## 7. 第二階段規格：解析與法規依據

### 7.1 功能清單

1. 作答後（或查看解析時）顯示：正確答案說明、各選項為何對／錯、對應法規條號。
2. 法規條號可點擊連結至全國法規資料庫對應條文（外部連結，新分頁開啟）。
3. 錯題本頁面直接顯示解析，方便使用者複習。
4. 解析內容標註參考來源（如「政府採購稽核發現缺失實例彙編」案例編號），供使用者查證。

### 7.2 內容製作流程與品質控管

解析與法規依據無法由既有 PDF 直接取得，需另行製作，建議流程：

1. 依題庫類別分批處理（建議每批 50–100 題），避免一次性製作品質不穩。
2. **AI 批次草擬**：撰寫離線批次腳本（Offline Batch Job），將每題的題幹、選項與正確答案（`answer`）餵給 AI（如 Claude），批次生成「正確答案理由」「其他選項錯誤原因」與對應的政府採購法／施行細則條號草稿，取代逐題人工從零撰寫。人力角色轉為「審核與微調」而非「撰寫」，可大幅提升第二階段產出速度。
3. 對照《政府採購稽核發現缺失實例彙編》與對照表，確認該題涉及的常見錯誤態樣與案例是否與 AI 草擬解析一致；如有出入，以官方彙編內容為準修正。
4. 草稿完成後，由使用者（或熟悉採購法之人員）逐批審核，確認法條引用正確、無過時條號（採購法歷經多次修正，需核對現行版本）。**AI 生成的法條引用有幻覺風險**（可能編造看似合理但實際不存在或錯誤的條號），審核時不可只看語句是否通順，須逐條對照全國法規資料庫現行條文核實。
5. 審核通過後併入 `questions.json` 的 `explanation` 與 `legal_refs` 欄位。

此流程屬於內容製作作業，工作量與題庫規模（約 3,600 題）成正比，建議先從高頻主題類別（如「政府採購法之總則、招標及決標」）開始，而非要求一次完成全部題庫。導入 AI 批次草擬後，工作量瓶頸應從「撰寫」轉移至「審核」，里程碑三驗證流程時應一併確認審核所需時間是否符合預期。

### 7.3 資料模型異動

沿用第 5.2 節 `explanation` / `legal_refs` / `source_note` 欄位；資料存取層新增 `getExplanation(questionId)`。

### 7.4 UI 異動

- 作答元件新增「查看解析」按鈕與展開區塊。
- 錯題本清單項目新增「解析」摘要預覽。

### 7.5 開發任務清單

1. 擴充 `questions.json` 結構與對應的資料存取層方法。
2. 實作解析展開元件（含法規連結）。
3. 建立內容審核用的檢查清單（法條版本、案例來源、格式一致性）。
4. 撰寫 AI 批次草擬腳本（讀取題幹／選項／答案，呼叫 AI 產出解析與法條引用草稿，輸出待審核格式）。
5. 依 7.2 節流程，分批完成題庫解析內容並匯入。
6. 抽樣覆核已上線的解析內容（建議每批完成後抽 10% 覆核）。

### 7.6 驗收標準

- 每題解析內容需附至少一則法規條號引用。
- 法規連結可正確導向對應條文頁面。
- 抽樣覆核未發現條號錯誤或明顯過時內容。

## 8. 第三階段規格：AI 對話問答＋搜尋

技術選型使用者表示暫不決定，本節先提供架構與選項比較，待後續確認後再細化。

### 8.1 功能清單

1. 問答輸入框：使用者以自然語言提問（如「巨額採購的門檻是多少」）。
2. 系統回覆：條文依據＋簡要說明，並標明引用來源（法條或稽核彙編案例）。
3. 全文搜尋：可直接搜尋題庫、法規條文、稽核案例中的關鍵字。
4. 問答與搜尋結果可連結回對應的練習題目（如有相關題目）。

### 8.2 技術選項比較

| 方案 | 優點 | 缺點 |
|---|---|---|
| Claude API＋檢索增強（RAG） | 回答品質高，可要求附引用來源，可涵蓋法規全文與稽核彙編 | 需 API 金鑰與後端服務，有使用費用 |
| 開源模型／規則式問答 | 成本低 | 覆蓋範圍與準確度有限，法律類問答風險較高 |

由於本專案內容屬法規教學，回答需要可追溯的條文引用，建議傾向 Claude API＋RAG 方案；但金鑰與費用管理需使用者確認後才進入開發。

### 8.3 架構設計要點

1. 建立法規與稽核彙編的檢索語料庫（依第 5.3 節結構切塊）。
2. 新增後端服務：接收前端問題 → 檢索相關法條／案例片段 → 組合提示詞呼叫 Claude API → 回傳答案與引用來源。
3. 前端僅呼叫自家後端，不直接接觸 Claude API 金鑰。

### 8.4 安全與成本考量

- API 金鑰僅存在後端環境變數，不進入前端程式碼或版本控制。
- 需設計基本的用量限制（如每日提問次數上限），避免費用失控。
- 回覆內容應附加免責提示（見第 9 節），避免使用者將練習網站回覆當作正式法律意見。

### 8.5 開發任務清單（待技術選型確認後展開細項）

1. 確認技術方案（Claude API 或其他）與費用預算。
2. 建立法規／稽核彙編檢索語料庫。

### 8.6 驗收標準（草案）

- 針對題庫涵蓋範圍內的問題，回覆需附至少一則可查證的法規或案例引用。
- 對超出題庫範圍或無法確認答案的問題，系統應明確告知「無法確認」而非臆測回答。

## 9. 非功能需求

1. **行動裝置相容**：練習頁面需支援手機瀏覽器操作，作答按鈕與選項需適應小螢幕。
2. **效能**：題庫載入（3,600 題規模）需採分類延遲載入，避免首頁一次載入全部題庫造成延遲。
3. **資料正確性**：第二階段解析內容上線前需經人工審核（見 7.2 節）；第三階段 AI 回覆需明確標示「僅供練習參考，非正式法律意見」。
4. **瀏覽器支援**：以現行主流瀏覽器（Chrome、Edge、Safari 最新兩版）為基準，不特別支援舊版 IE。
5. **可維護性**：法規會修法異動，`legal_refs` 需記錄條號對應的法規版本或查詢日期，方便日後盤點是否過時。

## 11. 風險與待確認事項

1. **題庫授權**（已確認）：`20260625全部題庫.pdf` 內容來源為行政院公共工程委員會「政府電子採購網」公開題庫查詢系統（<https://web.pcc.gov.tw/psms/plrtqdm/questionPublic/indexReadQuestion>），該系統提供「試題類別」查詢與「下載題庫」功能，14 個試題類別與本專案解析出的分類完全一致，屬工程會公開提供之官方題庫，非商業付費題庫，公開上線無授權疑慮。
2. **法條時效**：政府採購法與施行細則歷經多次修正，第二階段製作解析時需以現行版本為準，並定期複查是否修法。
3. **解析製作工作量**：約 3,600 題全部製作解析，工作量大，建議分批並可考慮先選定「常考／高頻」題目優先處理，而非強求一次到位。
4. **第三階段成本與範圍**：Claude API 用量費用、是否開放公眾使用或僅限使用者本人使用，將影響架構複雜度，待確認後再展開細部設計。
5. **部署方式**：目前確認先在本機開發測試，之後是否公開部署（GitHub Pages／自有伺服器）待後續決定，決定後需視情況調整第三階段後端部署方式。
