## 1. 目標

對應 [政府採購法練習網站_開發規格書v1.md](政府採購法練習網站_開發規格書v1.md) 第7.2節「內容製作流程與品質控管」：讓題庫（3,599題）每題都能附上「解析」（為何正確答案對、其他選項錯在哪）與「法規依據」（法條號、必要時附子法規/函釋），而不是只有答案。

## 2. 整體架構

```
題目（questions.json）
   │
   ├─ 有 source_legal_ref（944題，來自解析 PDF 時留下的「依據法源」欄位）
   │      │
   │      ▼
   │  law_lookup.py：依條號精準查政府採購法／施行細則全文
   │
   └─ 無 source_legal_ref（約2,650題）或 source_legal_ref 不足以完整解題
          │
          ▼
      corpus_search.py：跨檔案關鍵字搜尋（涵蓋52+份子法規、稽核彙編、
      預算編列手冊等，題庫本身排除以避免循環論證）
          │
          ├─ 找到依據 → 草擬解析＋法規依據，標記信心等級
          │
          └─ 查無依據 → 送 NotebookLM（瀏覽器自動化）查證
                 │
                 ├─ 找到 → 補上依據，信心等級調整為高
                 └─ 仍查無 → 標記為「超出目前語料範圍」，記錄需要補充的文件
```

## 3. 已建立的工具

### 3.1 `website/scripts/law_lookup.py`
解析政府採購法／施行細則 md 檔案，依條號（如「第22條」「第11-1條」）建立查找字典，回傳條文全文。適用於已知 `source_legal_ref` 的題目，精準、免關鍵字誤判。

**重要限制**：施行細則的條號是**獨立編號**，與母法條號不是同一主題（例如施行細則第94條講「30日內辦理驗收」，跟母法第94條「採購評選委員會」完全無關，純屬號碼巧合）。**不能假設「母法第N條」和「施行細則第N條」講同一件事**，每次都要另外用關鍵字確認施行細則裡的真正對應條號。這在18題試跑與45題批次中都各出現過一次，是系統性問題，全量生產時要特別注意。

### 3.2 `website/scripts/corpus_search.py`
跨檔案關鍵字搜尋工具，涵蓋 `法規/` 資料夾下所有 md 檔案（自動排除題庫本身，避免拿題庫的敘述佐證題庫自己的答案）。用於：
- 沒有 `source_legal_ref` 的題目找依據。
- `source_legal_ref` 指向的母法條文不足以解釋題目具體數字/細節時（通常細節藏在子法規或稽核彙編案例裡）。

### 3.3 NotebookLM 瀏覽器自動化查證（備援管道）
用 `claude-in-chrome` 工具操作使用者的 NotebookLM notebook（<https://notebooklm.google.com/notebook/dc20365f-3f3d-447f-abbe-ad71d1147d9f>，已含52+份法規來源）。適用時機：`law_lookup.py` 與 `corpus_search.py` 都查無依據時的最後備援，**不是每題都要用**，用意是降低 NotebookLM 的查詢量，只處理真正卡住的題目。

操作要點：
- 點擊聊天輸入框 → 打字 → **點藍色送出鈕**（Enter 鍵在這個輸入框只會換行，不會送出，這是實測發現的坑）。
- 送出後约需等待 10-20 秒生成回覆，需視情況多次等待+截圖確認完成。
- 已實測驗證兩次，回覆品質與交叉核對結果一致，可信賴作為備援。

## 4. 信心等級標記

每題草擬完解析後標記三級信心，供人工審核時快速抓重點：

- 🟢 **高**：條文/子法規逐字比對，論證直接。
- 🟡 **中**：找到明確依據方向，但細節（人數、比例、期限等）未逐字查證原文條號，或依常理／間接推論判斷，建議人工覆核。
- 🔴 **低／查無依據**：語料庫中找不到直接依據，需要標注「超出目前語料範圍」，不可硬編答案冒充有依據。

**實測數據**：18題試跑最終16→18題達🟢（3題原本卡住，皆透過使用者協助覆核/補件解決）；45題批次38題🟢、7題🟡、原1題🔴已透過NotebookLM解決為🟢。

## 5. 效率技巧（來自45題批次驗證）

3,600題中很多題目共用同一法條依據（例如單是第22條就有154題），**同一法條分組處理**可大幅提效：第一題查完整條文後，同組後續題目可直接套用同一段落做論證，不需要每題重新檢索。全量生產應優先按 `source_legal_ref` 出現頻率排序處理，高頻法條優先，效益最大。

## 6. 語料庫現況與擴充紀錄

初始 52 份法規／子法規（使用者用 markitdown 轉檔），後續依試跑過程發現的缺口陸續補充：

| 補充項目 | 補充原因 |
|---|---|
| 《公共建設工程經費估算編列手冊（92年版）》 | lifecycle-tf-016 涉及工程預算科目定義，原52份沒有 |
| 《統包作業須知》 | 統包相關細節補強 |

**已知但尚未補充的候選來源**：
- **政府採購法規解釋函令及相關函文查詢**（<https://planpe.pcc.gov.tw/prms/explainLetter/readPrmsExplainLetterSearch>，使用者提供）——工程會官方函釋資料庫。許多🟡信心題目（人數門檻、期限細節、程序性判斷）很可能答案依據不在法條本身，而在行政函釋裡，這是下一步最值得補充的語料來源，可望把45題批次裡的7題🟡直接轉🟢。
- 《建築法》《建築物室內裝修管理辦法》等目的事業法規——僅在涉及跨領域題目（如室內裝修設計）時需要，優先度較低，且已證實可透過稽核彙編案例間接取得依據（見試跑第15題經驗），非必要立即補充。

## 7. 尚未解決的開放問題（留待下次討論）

1. **函釋資料庫怎麼整合**：是要下載轉檔成 md 併入語料庫，還是即時查詢 PCC 網站？兩種方式的檢索工具設計不同。
2. **排程執行的具體設計**：權限模式設定（避免半夜卡在確認提示）、prompt 內容（自我完備，不依賴當前對話上下文）、進度追蹤機制（如何記錄已完成/未完成題目，讓多次排程執行可以接續而不重工或漏題）。
3. **輸出資料結構定案**：目前試跑/批次文件是人工審閱用的 md 格式；正式併入 `questions.json` 的 `explanation`／`legal_refs`／`source_note` 欄位（依規格書5.2節結構）需要再設計一次轉換腳本。
4. **🟡信心題目的覆核順序**：是先全部跑完一輪（3,600題）再統一覆核🟡🔴項目，還是每個批次跑完就先覆核再繼續下一批。

## 8. 使用者對品質的信心與定位

使用者已明確表示對 Claude＋NotebookLM 交叉驗證的方法論有高度信心，主要疑慮不是準確度而是**語料完整度**（法規/函釋是否齊全）。這代表未來工作重心應放在：(1) 持續擴充語料庫（尤其函釋資料庫），(2) 維持現有的信心分級與交叉驗證機制，而非改變核心方法論。
